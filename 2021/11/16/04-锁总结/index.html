事务并发访问同一条记录的情况

  • 读-读、读-写、写-读、写-写

写-写：通过为记录加锁，让它们排队执行

写写的情况

  • 脏写
事务与锁结构

当一个事务T1想对这条记录进行改动时，首先会看看内存中有没有与这条记录关联的锁结构，如果没有就会生成一个锁结构与之关联。这种场景称为加锁成功。

此时另一个事务T2也想对该记录进行改动时，在发现内存中有与该记录关联的锁结构，于是也生成了一个锁结构与该记录关联，只不过锁结构的is_waiting为true，也就是事务需要等待，不可以继续执行操作，这种场景我们称为加锁失败。

当事务T1提交或者回滚后，就会把它生成的锁结构释放掉，然后检测下还有没有与该记录关联的锁结构，结果发现事务2还在等待获取锁，就把T2事务对应的锁结构的is_waiting改为false，并唤醒事务T2的线程让它继续执行。

  • trx信息：表示这个锁结构与哪个事务关联的
  • is_waiting：表示当前事务是否在等待

读-写，写-读的情况

  • 会出现脏读、不可重复读、幻读的现象

怎么避免？

  • 方式一：读操作使用多版本并发控制MVCC，写操作进行加锁：读-写操作并不冲突，性能更高
  • 方式二：读操作、写操作都加锁：读写操作，需要排队，影响性能。

一致性读

在RC和RR隔离级别下：普通的select语句都是一致性读(快照读)，一致性读不会对表中的数据加锁

锁定读

锁定读：在读取该记录前就为该记录加锁的方式。

在使用方式二解决问题时，为了读-读之间不受影响，而读-写，写-读，写-写又相互阻塞。MySQL大叔给锁进行了分类

  • 共享锁(s)：在事务要读取一条记录时，需要先获取该记录的S锁
  • 排他锁(x)：在事务要修改一条记录时，需要先获取该记录的X锁
兼容性 S锁 X锁
S锁 兼容 不兼容
X锁 不兼容 不兼容

注：以上的锁就是为了在读-读的时候不受影响，在读-写，写-读，写-写的情况相互阻塞。

锁定读的语句
  • 对读取的记录加S锁：select … lock in share mode;
  • 对读取的记录加X锁：select … for update; 或者改动这些记录(DML)
写操作
  • DELETE
  • UPDATE
  • INSERT：一般新插入的一条记录，受隐式锁保护，不需要在内存生成锁结构。(也有特殊情况会在内存中生成锁结构)

注：一个事务加的锁，一般在事务提交或者中止时才会释放，也有特殊情况。

多粒度锁

  • 行级锁：针对记录加锁

  • 表级锁：针对表加锁

表级锁也可以分为，共享锁(S)和排他锁(X)

兼容性 S锁 X锁
S锁 兼容 不兼容
X锁 不兼容 不兼容

意向共享锁(IS)：当事务准备在某条记录上加S锁时，需要先在表级别加一个IS锁

意向排它锁(IX)：当事务准备在某条记录上加X锁时，需要先在表级别加一个IX锁

注：IS，IX是表级锁，它的提出仅仅是为了在之后加表级别的S锁和X锁时可以快速判断表中的记录没有没上锁。以避免用遍历的方式来检查。

IS锁和IX锁是兼容的：就是你对表中的某一行记录加的是S锁，对另一行记录加的也是X锁，此时加S锁时会在表上加一个IS锁，加X锁时会在表上加IX锁

IX锁和IX锁兼容的：就是你对表中的某一行记录加的是X锁，对另一行记录加的也是X锁，这两次都会往表上加IX锁。

InnoDB存储引擎中的锁

InnoDB中的表级锁

1、表级别的S锁、X锁

在对某个表执行CRUD时，innoDB存储引擎不会为这个表添加表级别的S锁或者X锁的

  • 为表加S锁：lock tables t read;

  • 为表加X锁：lock tables t write

注：关于表级别的S锁和X锁了解下即可。重点了解行级锁

2、表级别的IS锁，IX锁

3、表级别的AUTO-INC锁

InnoDB中的行级锁

行级锁：顾名思义在记录上加的锁

1、Record Lock：仅仅把一条记录锁上

2、GAP Lock（间隙锁）：

由来：在MySQL的RR隔离级别下，很大程度解决幻读的方案有两种，1是使用MVCC，2是使用加锁。
这里如果使用加锁的方式，会有一个问题，就是事务在第一次读取的时候，那些幻影记录不存在，无法给这些幻影记录加上锁，因此提出了GAP锁

作用：就是为了防止插入幻影记录而已。

注：给一条记录加gap锁，只是不允许其他事务向这条记录前面的间隙插入新记录。有两个疑问怎么确定该记录加的是gap锁，还有就是为什么只是不允许在前面的间隙插入，在后面的间隙插入就允许了？不是说可以可以防止幻影记录吗，我加到后面还不是防止不了？

3、Next-Key Lock（临键锁）

有时候，既想锁住某条记录，又想阻止其他事务在该记录前面的间隙插入新记录。

next-key lock = gap lock + record lock;

4、Insert Intention Lock（插入意向锁）

事务在插入一条记录前，需要判断插入位置是否被别的事务加了gap锁，如果加了，就需要等待，并且在内存中

生成一个锁结构，我们管这种锁叫：插入意向锁

注：插入意向锁之间不会相互阻塞

注：一般是INSERT语句

5、隐式锁

一个事务对新插入的记录可以不显示的加锁(生成一个锁结构)，如果别的事务对这条记录加S锁或者X锁时，由于隐式锁的存在，会帮当前事务的记录生成一个锁结构。

InnoDB锁的内存结构

对一条记录加锁的本质就是在内存中创建了一个锁结构与之关联（隐式锁除外）。

对不同记录加锁时如果符合下面的条件，这些记录的锁就可以共用一个锁结构

  • 在同一个事务中进行加锁操作
  • 被加锁的记录在同一个页面中
  • 加锁的类型是一样的
  • 等待的状态是一样的
InnoDB中的锁结构
  • 锁所在的事务信息
  • 索引信息：对于行级锁来说，记录一下加锁的记录是属于哪个索引
  • 表锁/行锁信息：行锁的三个信息：Space ID 表空间，Page Number 记录所在的页号，n_bits：对于行锁，一个比特对应一条记录
  • type mode
    • lock_mode：是IS，IX等
    • rec_lock_type：行锁的具体类型
    • lock_type：锁类型：是表级锁，还是行级锁

语句加锁分析

1、普通的select语句

不同的隔离级别，普通的select语句具有不同的表现

  • 读未提交：不加锁，直接读取版本链中的最新记录，可能会出现脏读，不可重复读，幻读
  • 读已提交：不加锁，每次执行普通select语句，都会生成readview，这样可以避免脏读
  • 可重复读：不加锁，只在第一次执行普通的select语句生成readview，这样可以把脏读，不可重复读，幻读（不能完全避免）避免了
  • 串行化
    • 手动提交事务：普通的select语句，转换为select…lock in share mode
    • 自动提交事务：不加锁

注：MVCC不能完全禁止幻读，在MySQL的RR隔离级别中，很大程度避免幻读，而不是完全避免！！！

2、锁定读语句

  • select … lock in share mode;
  • select … for update;
  • update …
  • delete …

注：对于update，delete语句来说，在对被更新或者被删除的二级索引记录加锁，实际上加的是隐式锁

注：<=RC，采用的是半一致性读的方式来执行update语句

匹配模式：

  • 如果是单点扫描区间，匹配模式就是精确匹配

唯一性搜索：

  • 匹配模式为精确匹配
  • 使用的索引是主键或者唯一二级索引
影响记录加锁的因素
  • 事务的隔离级别
  • 语句执行时使用的索引类型（聚簇索引，唯一二级索引，普通二级索引）
  • 是否是精确匹配
  • 是否是唯一性搜索
  • 具体执行语句的类型（insert，delete，update，select）

注：锁的释放一般是通过事务的提交或回滚，在<RC隔离级别下，可能也会提前将不符合搜索条件的记录上的锁给释放掉。

注：我们把锁定读，看成是依次读取若干个扫描区间中的记录

  • 步骤1：快速在B+树的叶子节点定位到扫描区间中的第一条记录，把该记录作为当前记录

  • 步骤2：为当前记录加锁，<=RC隔离级别加record lock，>=RR隔离级别加next-key lock;

  • 步骤3：判断索引条件下推的条件是否成立，适于于二级索引和select语句，如果符合跳到步骤4，不符合：就获取当前记录的下一条记录，然后跳到步骤2 ，还会判断是否符合边界条件，如果不符合，直接向Server层返回一个查询完毕的信息，这里不会释放锁！！

  • 步骤4：执行回表操作，如果是二级索引，则需要进行回表操作，获取到对应的聚簇索引记录并给它加record lock

  • 步骤5：判断边界条件是否成立：如果不符合：<=RC隔离级别就得释放在该记录上的锁，如果是>=RR隔离级别不会释放锁

  • 步骤6：server层判断搜索条件是否成立：成立就把记录发送到客户端，不成立，<=RC释放锁，>=RR不释放锁

  • 步骤7：获取当前记录所在单向链表的下一条记录，并将它作为新的记录，跳到步骤2。

注：边界条件：把形成扫描区间的搜索条件称为边界条件。

注：不是说在>=RR隔离级别下，一定是加next-key lock，只能说一般情况是这样，也有特殊情况是加gap锁或者record锁！！！

在RR隔离下，使用的是主键索引，不是精确匹配和唯一性的搜索，执行的语句类型是select

hero表全部记录主键如：1，3，8，15，20

select * from hero where number > 3 and number < 8 for update

这个时候扫描区间是：(3, 8)，但是加锁会把3，8，15的记录加上。

加锁步骤的流程图
  • 读取扫描区间的第一条记录
  • 为当前记录加锁，一般如果是<=RC隔离级别加的是record lock，如果是>=RR隔离级别加的是next-key lock
  • 是否符合索引下推的条件
  • 是否需要执行回表：二级索引记录会执行，并且为对应的聚簇索引加一个S锁record lock锁
  • 是否符合边界条件
  • 是否符合搜索条件

演示时需要用到的SQL

<=RC隔离级别加锁的情况（SQL语句使用的是聚簇索引）

SQL：select * from hero where number > 1 and number <= 15 and country = ‘魏’ lock in share mode;

总结：根据加锁的规则，只为最终的结果集记录加record lock；

>=RR隔离级别加锁的情况（SQL语句使用的是聚簇索引）

步骤1：定位到扫描区间的第一条记录，也就是这里的number = 3的记录

步骤2：加锁，一般来说对<=RC加record lock，>=RR加next-key lock

步骤3：索引下推的条件是否成立：聚簇索引不需要索引下推

步骤4：是否需要执行回表操作：聚簇索引不需要回表

步骤5：是否满足边界条件：符合

步骤6：是否满足搜索条件：不符合并且隔离级别>RR，所以不释放锁

一直走到number=15，然后获取下一条记录是number = 20，从步骤2开始：加完了锁，由于是聚簇索引不需要判断索引下推和回表的操作，所以跳过步骤3和步骤4

到步骤5的时候，不满足边界条件，由于>=RR所以不释放锁，并向Server层返回一个”查询完毕”的信息

总结：只执行SQL中边界条件过滤下来的聚簇索引记录 和 扫描区间后面下一条记录加next-key锁

<=RC隔离级别加锁的情况（SQL语句使用的是二级索引）

加锁步骤：

步骤1：读取扫描区间的第一条记录，也就是number=1的刘备

SQL语句：select * from hero where name > ‘c曹操’ and name <= ‘x荀子’;

步骤二：给记录加锁，一般来说<=RC加record lock，>=RR加next-key lock;

步骤三：explain select * from hero force index(idx_name) where name > ‘c曹操’ and name <= ‘x荀子’ and country != ‘吴’ lock in share mode; 由于走的是二级索引，需要判断是否符合索引下推的条件，这里number=1的记录符合条件，就跳到的步骤四

步骤四：给二级索引执行回表操作，去聚簇索引中获取主键值为1的索引，给加上S型record lock;

步骤五：符合边界条件，跳转到步骤六

步骤六：符合搜索条件，将该记录发送到客户端

步骤七：根据链表获取当前记录的下一条记录，也就是number=2的孙权，然后执行步骤二加锁，直到不满足步骤6，又由于是<=RC，所以锁被释放了。

怎么看扫描区间后面下一条记录是什么？

总结：最终的结果集二级索引记录和对应的聚簇索引记录加record lock，扫描区间后面下一条记录加record lock锁

注：因为它是二级索引执行了步骤三，由于在步骤二加完的锁，跑到步骤三的时候不符合条件所以直接像service层发送一个查询完毕的信息，注意但是不会释放锁。这也就是这里扫描区间后面下一条记录加了锁的原因，而上面<=RC隔离级别，使用的是聚簇索引。扫描区间的下一条记录没有加锁的原因是，因为是聚簇索引不需要执行步骤三、步骤四。执行到步骤5不符合边界条件，又由于是<=RC隔离级别，所以锁就被释放掉了。

>=RR隔离级别加锁的情况（SQL语句使用的是二级索引）

步骤1：定位到扫描区间的第一条记录，也就是这里的name= ‘c曹操’的记录

步骤2：加锁：一般来说对<=RC加record lock，>=RR加next-key lock

步骤3：索引下推的条件是否成立：符合

步骤4：是否需要执行回表操作：需要，为对应的聚簇索引记录加上S型record lock

步骤5：是否满足边界条件：符合

步骤6：是否满足搜索条件：不符合，由于>=RR所以不会释放锁

总结：扫描区间中的记录，不管满不满足搜索条件，都会加上next-key锁，对应的聚簇索引记录加S型的record lock锁，扫描区间后面的下一条记录加next-key锁

总结：一般情况下加锁
  • 在<=RC隔离级别，并且使用的是聚簇索引：就为SQL执行后的最终的聚簇索引记录加record lock锁
  • 在<=RC隔离级别，并且使用的是二级索引：就为SQL执行后的最终的二级索引记录加record lock和对应的聚簇索引记录加record lock，还有扫描区间后面的下一条记录加record锁
  • 在>=RR隔离级别，并且使用的是聚簇索引：就为只把边界条件作为搜索条件的SQL执行后的过滤下来的聚簇索引记录都加next-key lock还有还有扫描区间后面的下一条记录加next-key lock锁
  • 在>=RR隔离级别，并且使用的是二级索引：就为只把边界条件作为搜索条件的SQL执行后的过滤下来的二级索引记录加next-key锁和对应的聚簇索引记录加S型record lock锁，还有二级索引扫描区间后面的下一条记录加next-key lock锁
总结：特殊情况下加锁
  • <=RC隔离级别，索引为二级索引，匹配模式为精确匹配（一般指单点扫描区间）

那么只为二级索引记录和对应的聚簇索引记录加record lock，至于是什么类型的record lock取决于你使用的语句

比如：使用的是select…lock in share mode; 那么就是S型的record lock，如果是select … for update那么就是X型的record lock

  • 在>=RR隔离级别，索引为二级索引，匹配模式为精确匹配（一般指单点扫描区间）

那么为二级索引记录加gap锁，对应的聚簇索引记录加record lock，扫描区间后面的下一条记录加gap锁

有时扫描区间没有记录，那也要扫描区间后面的下一条记录加一个gap锁

  • 在>=RR隔离级别，匹配模式不是精确匹配，并且没有找到匹配的记录，则会为该扫描区间后面的下一条记录加next-key锁
  • 在>=RR隔离级别，索引为聚簇索引，并且扫描的扫描区间是左闭区间。
  • 无论是哪个隔离级别，只要是唯一性搜索，就为该记录添加正经记录锁
